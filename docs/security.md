# セキュリティと権限設計

## 認証

- Firebase Authentication (メール/パスワード) を使用
- バックエンドで ID トークン検証 (`firebase-auth-cloudflare-workers`)
- メール認証済みのみ許可 (`email_verified: true`)
- 未認証: 401、メール未認証: 403 (`EMAIL_VERIFY_MISSING`)
- トークンは 10 分ごとにフロントエンドで自動更新

## グループロール

| ロール  | 権限                                       |
| ------- | ------------------------------------------ |
| `ADMIN` | 全権限 (メンバー管理、スカウト削除、移動)  |
| `EDIT`  | スカウト作成/編集/共有、閲覧               |
| `VIEW`  | 閲覧のみ                                   |

ロールは `auth.memberships` に `"ROLE;groupId"` 形式で保持 (最大10件)

### ロール継承

```
ADMIN > EDIT > VIEW
```

ADMIN は EDIT の全権限を含み、EDIT は VIEW の全権限を含む

## スカウト共有ロール

| ロール  | 権限             |
| ------- | ---------------- |
| `EDIT`  | 閲覧 + 編集      |
| `VIEW`  | 閲覧のみ         |

`auth.shares` に `scoutId` 形式で保持 (最大10件)

## 権限制御の詳細

### スカウト操作

| 操作     | 必要権限                                           |
| -------- | -------------------------------------------------- |
| 検索     | グループメンバー                                   |
| 取得     | グループメンバー **または** share                  |
| 作成     | グループ `ADMIN` または `EDIT`                     |
| 更新     | グループ `ADMIN`/`EDIT` **または** share `EDIT`    |
| 削除     | グループ `ADMIN` のみ                              |
| 移動     | グループ `ADMIN` のみ                              |
| 共有管理 | グループ `ADMIN` または `EDIT`                     |

### グループ操作

| 操作           | 必要権限 |
| -------------- | -------- |
| プロファイル取得 | メンバー |
| 招待一覧       | `ADMIN`  |
| 招待作成       | `ADMIN`  |
| メンバー一覧   | `ADMIN`  |
| メンバー削除   | `ADMIN`  |
| ロール変更     | `ADMIN`  |

### ユーザー操作

- 自分自身のレコードのみ操作可能
- 他ユーザーの検索は `lookupByEmail` (メールで検索)

### God モード

- `auth.isGod === true` のユーザーのみ利用可能
- すべての認可チェックをバイパス
- スカウト/グループ/ユーザーの直接操作が可能
- システム管理用途

## 認可チェックの実装

```typescript
// グループロール確認
c.var.user.fn.isInRoleOnGroup(groupId, ["ADMIN", "EDIT"])

// 共有確認
user.auth.shares.includes(scoutId)

// God確認
user.auth.isGod === true
```

## セキュリティ対策

### 入力検証 (Zod)

- ID: `[A-Za-z0-9_-]+` (1-100文字)
- 日付: `YYYY-MM-DD` (実在日付チェック)
- メール: RFC 5322 準拠
- 配列上限: memberships/invites/shares は最大10件

### 自己参照防止

- 自分を招待/共有不可
- 自分のロール変更不可
- 自分をグループから削除不可

### 重複防止

- 既存メンバーへの再招待防止
- 既存共有の重複防止

### 上限チェック

- 招待上限: 10件/ユーザー
- メンバーシップ上限: 10件/ユーザー
- 共有上限: 10件/ユーザー

## Firestore ルール

- バックエンドは Service Account で Firestore にアクセスするため、ルールは API 経由では適用されない
- 認可はすべてバックエンドのミドルウェアとハンドラーで実施
- Firestore ルールは直接アクセス時の安全策として設定可能

## 既知の制限

- レート制限未実装
- 監査ログ未実装
- 開発環境 (`IS_DEV === "TRUE"`) では CORS が `*` で許可される
