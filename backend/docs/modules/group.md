# Group モジュール仕様

## 概要

グループ(団)の管理機能を提供するモジュール
グループの作成・更新・削除、メンバー管理、権限管理を行う

## ディレクトリ構造

```
group/
├── groupRoute.ts              # [API層] HTTPエンドポイント定義
├── services/                  # [ビジネスロジック層]
│   └── groupService.ts        # ビジネスロジック
└── permissions/               # [権限チェック層]
    └── groupPermissions.ts    # 権限判定ロジック
```

## 各層の責務

### API 層

**groupRoute.ts**

- HTTP エンドポイントの定義とルーティング
- リクエストのバリデーション(Zod)
- サービス層への処理委譲
- レスポンスの返却

**エンドポイント:**

- GET /:id - グループ取得
- POST / - グループ作成
- PUT /:id - グループ更新
- DELETE /:id - グループ削除
- POST /:id/members - メンバー追加
- DELETE /:id/members/:email - メンバー削除
- PUT /:id/members/:email/role - メンバーロール更新

### ビジネスロジック層

**services/groupService.ts**

- グループ操作のビジネスロジック実装
- メンバー管理のビジネスロジック実装
- 権限チェック層と Firestore 操作層の橋渡し
- エラーハンドリングと HTTP 例外の生成

**主要な関数:**

- `getGroupWithAuth()` - グループ取得(メンバー確認付き)
- `createGroup()` - グループ作成(作成者が ADMIN になる)
- `updateGroupWithAuth()` - グループ更新(ADMIN 権限必須)
- `deleteGroupWithAuth()` - グループ削除(ADMIN 権限必須)
- `addGroupMember()` - メンバー追加(ADMIN 権限必須)
- `removeGroupMember()` - メンバー削除(ADMIN 権限必須)
- `updateMemberRole()` - メンバーロール更新(ADMIN 権限必須)

### 権限チェック層

**permissions/groupPermissions.ts**

- グループへのアクセス権限の検証
- グループ管理操作の権限判定
- 管理者権限の検証

**主要な関数:**

- `isGroupAdmin()` - ADMIN 権限の確認
- `isGroupMember()` - メンバーシップの確認
- `canUpdateGroup()` - 更新権限の確認
- `canDeleteGroup()` - 削除権限の確認
- `canManageGroupMembers()` - メンバー管理権限の確認

## データモデル

### GroupRecord

```typescript
{
  name: string                 // グループ名
  status: "ACTIVE" | "INACTIVE"  // ステータス
  members: [                   // メンバー配列
    {
      userEmail: string        // ユーザーのメールアドレス
      role: "ADMIN" | "EDIT" | "VIEW"  // ロール
    }
  ]
}
```

### GroupMember

```typescript
{
  userEmail: string; // ユーザーのメールアドレス
  role: "ADMIN" | "EDIT" | "VIEW"; // ロール
}
```

## ロール(権限レベル)

### VIEW (閲覧)

**できること:**

- グループ情報の閲覧
- スカウト情報の閲覧
- グループ内スカウトの検索

**できないこと:**

- スカウトの作成・編集・削除
- グループ設定の変更
- メンバー管理

### EDIT (編集)

**できること:**

- VIEW の権限に加えて
- スカウトの作成・更新
- 自分のプロフィール編集

**できないこと:**

- スカウトの削除
- グループ設定の変更
- メンバー管理

### ADMIN (管理者)

**できること:**

- EDIT の権限に加えて
- スカウトの削除
- グループ設定の変更
- メンバーの追加・削除
- メンバーのロール変更
- グループの削除

**責任:**

- グループ内のデータ管理
- メンバーの権限管理
- 最低 1 人の ADMIN を維持

## ビジネスルール

### グループ作成

**処理:**

1. グループ ID をランダム生成
2. グループレコード作成
3. 作成者を自動的に ADMIN ロールで追加
4. 作成者のユーザー情報に`joinedGroupId`を設定

**条件:**

- 認証済みユーザーのみ実行可能
- グループ ID は自動生成

### グループ更新

**処理:**

1. グループの存在確認
2. ADMIN 権限チェック
3. グループ情報更新

**条件:**

- ADMIN 権限が必要
- 自分が所属するグループのみ更新可能

**更新可能なフィールド:**

- `name` - グループ名
- `status` - ステータス

### グループ削除

**処理:**

1. グループの存在確認
2. ADMIN 権限チェック
3. メンバー数チェック(1 人のみであること)
4. グループレコード削除

**条件:**

- ADMIN 権限が必要
- メンバーが 1 人のみの場合のみ削除可能
- 複数メンバーがいる場合はエラー

**理由:**

- データの誤削除防止
- メンバー全員の同意が必要
- 先にメンバーを削除してから実行

### メンバー追加

**処理:**

1. グループの存在確認
2. ADMIN 権限チェック
3. メンバーの重複チェック
4. メンバーリストに追加

**条件:**

- ADMIN 権限が必要
- 追加するユーザーのメールアドレスが必要
- 既存メンバーでないこと

### メンバー削除

**処理:**

1. グループの存在確認
2. ADMIN 権限チェック
3. 削除対象メンバーの存在確認
4. 最後の ADMIN チェック
5. メンバーリストから削除

**条件:**

- ADMIN 権限が必要
- 最後の ADMIN は削除不可
- 削除後も ADMIN が 1 人以上残る必要がある

**理由:**

- グループ管理者不在を防止
- 最低 1 人の ADMIN を維持

### メンバーロール更新

**処理:**

1. グループの存在確認
2. ADMIN 権限チェック
3. 更新対象メンバーの存在確認
4. 最後の ADMIN チェック(ADMIN→ 他のロール変更時)
5. ロール更新

**条件:**

- ADMIN 権限が必要
- 最後の ADMIN のロール変更は不可
- 変更後も ADMIN が 1 人以上残る必要がある

## メンバーシップ管理

### ユーザーとグループの関係

- ユーザーは 1 つのグループにのみ所属可能
- グループ作成時に自動的に所属グループが設定される
- `User.joinedGroupId`とグループメンバーリストの両方で管理

### メンバー情報の同期

**グループ作成時:**

1. グループレコードのメンバーリストに作成者追加
2. ユーザーレコードの`joinedGroupId`に設定

**メンバー追加時:**

1. グループレコードのメンバーリストに追加
2. ユーザーレコードの`joinedGroupId`に設定

**メンバー削除時:**

1. グループレコードのメンバーリストから削除
2. ユーザーレコードの`joinedGroupId`をクリア

## エラーハンドリング

### 発生する可能性のある HTTPException

| ステータス    | 説明                                 | 発生条件                              |
| ------------- | ------------------------------------ | ------------------------------------- |
| 404 Not Found | グループまたはメンバーが見つからない | 存在しない ID・メールアドレス         |
| 403 Forbidden | 権限がない、またはビジネスルール違反 | ADMIN 権限不足、最後の ADMIN 削除など |
| 409 Conflict  | メンバーが既に存在する               | メンバー追加時の重複                  |

## セキュリティ

### 認証

- すべてのエンドポイントで Firebase 認証が必須
- ID トークンの検証を実施

### 認可

- グループ情報の閲覧: メンバーのみ
- グループの更新・削除: ADMIN のみ
- メンバー管理: ADMIN のみ

### データ保護

- グループメンバーのみがグループデータにアクセス可能
- メンバー以外はグループの存在を確認できない
- ADMIN 権限による厳格な管理

## 処理フロー

### グループ作成フロー

```
1. API層: リクエスト受付とバリデーション
   ↓
2. Service: ランダムID生成
   ↓
3. Service: グループレコード作成(作成者をADMINで追加)
   ↓
4. Operations: Firestoreに保存
   ↓
5. Service: ユーザーのjoinedGroupId更新
   ↓
6. API層: レスポンス返却
```

### グループ更新フロー

```
1. API層: リクエスト受付とバリデーション
   ↓
2. Service: グループ取得と存在確認
   ↓
3. Permissions: ADMIN権限チェック
   ↓
4. Operations: Firestoreを更新
   ↓
5. API層: レスポンス返却
```

```

   ↓

3. Permissions: ADMIN権限チェック
   ↓
4. Service: メンバー数チェック(1人のみ)
   ↓
5. Operations: Firestoreから削除
   ↓
6. API層: レスポンス返却
```

   ↓

   ↓

## 注意事項

### 最後の ADMIN 保護

- 最後の ADMIN は削除不可
- 最後の ADMIN のロール変更は不可
- 常に 1 人以上の ADMIN を維持

### メンバー削除前の確認

- グループ削除時はメンバーが 1 人のみであること
- 複数メンバーがいる場合は先にメンバー削除が必要

### ステータスの使い分け

- `ACTIVE`: 活動中のグループ
- `INACTIVE`: 休止中のグループ
- 機能的な制限はなく、表示上の区別のみ
